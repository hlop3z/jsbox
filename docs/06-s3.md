# 6. `s3` — Signed Upload & Download Links 🔗

[← Back to the guide](README.md)

`s3` makes a **special link** that lets a web page upload a file straight to your
storage bucket (S3, Cloudflare R2, MinIO, Backblaze B2…) — without the file ever
passing through jsbox.

## The big idea 💡

The robot does **not** touch your files. It only does some quick math to make a
**signed link** (a normal web address with a secret signature on the end). You give
that link to your web page, and the browser uploads the file directly. Fast, and the
robot never sees the bytes.

The robot never connects to your storage — it just does quick math and hands back a
link. It does **no file reading or writing** at all.

> 🛡️ **Safety rules (same as `api`).** The `endpoint` must start with `http://` or
> `https://` — never `file://`. And it must be a **public** address: `localhost`,
> `127.0.0.1`, and private network addresses (`10.x`, `192.168.x`, …) are **blocked**,
> so a link can never point at something local or internal. Your script can't change
> the `endpoint` either — only the `config` you send can.

## Turn it on first 🔑

Give the robot your bucket details with `config.s3`:

```json
{
  "config": {
    "s3": {
      "endpoint": "https://ACCOUNT.r2.cloudflarestorage.com",
      "region": "auto",
      "bucket": "uploads",
      "access_key": "AKID...",
      "secret_key": "SECRET...",
      "path_style": false,
      "max_upload_size": "25mb"
    }
  }
}
```

| Setting           | What it means                                               | Default    |
| ----------------- | ----------------------------------------------------------- | ---------- |
| `endpoint`        | Your storage address, starting with `http://` or `https://` | (must set) |
| `region`          | The storage "region" name (R2 uses `auto`)                  | (must set) |
| `bucket`          | The bucket (folder) name                                    | (must set) |
| `access_key`      | Your storage username                                       | (must set) |
| `secret_key`      | Your storage password                                       | (must set) |
| `path_style`      | `true` for MinIO/local; `false` for AWS-style `bucket.host` | `false`    |
| `expires`         | How long a link works, in seconds                           | `900`      |
| `max_expires`     | The longest a link may ever last (7 days)                   | `604800`   |
| `max_upload_size` | Biggest file `presignPost` will allow (`"25mb"`, `"50gb"`)  | (unset)    |

No `config.s3` → `s3` is turned off.

> **Which `path_style`?** MinIO and most self-hosted stores need `true`. AWS S3 and
> Cloudflare R2 use the default `false`.

> 🧪 **Testing with a local MinIO?** Because `localhost` is blocked by the safety rule
> above, set top-level `"debug": true` in the **server** config (`config.json`) to allow
> local addresses while you develop. Turn it **off** in production.

## Make an upload link: `s3.presignPut` ⬆️

```js
function handler(ctx) {
  var put = s3.presignPut({
    key: "uploads/" + ctx.filename, // where to save it in the bucket
    expires: 300, // link works for 5 minutes
  });
  // put = { url: "https://...&X-Amz-Signature=...", method: "PUT", expires: 300 }
  return json({ uploadTo: put.url }, null);
}
```

Your web page then uploads the file with that link:

```js
// in the browser:
await fetch(put.url, { method: "PUT", body: theFile });
```

## Make a download link: `s3.presignGet` ⬇️

```js
var get = s3.presignGet({ key: "uploads/photo.jpg" });
// get.url is a short-lived link to view/download the file
```

## Upload with a size limit: `s3.presignPost` 📏

`presignPut` lets the browser upload a file of **any** size. If you rent storage and want
to stop people uploading huge files, use **`presignPost`** instead. It makes a link with a
built-in **size limit** that the storage itself enforces — too-big uploads are rejected.

The size limit comes from your **config** (`max_upload_size`), **not** from the script.
That way a script can never raise it.

```json
{ "config": { "s3": { "...": "...", "max_upload_size": "25mb" } } }
```

```js
function handler(ctx) {
  // the cap is config.s3.max_upload_size — the script just picks the key
  var up = s3.presignPost({ key: "customers/" + ctx.id + "/" + ctx.name });
  // up = { url, fields: {...}, maxBytes: 26214400, expires: 900 }
  return json(up, null);
}
```

Your web page sends the file as a form (the `file` part goes **last**):

```js
// in the browser:
const form = new FormData();
Object.entries(up.fields).forEach(([k, v]) => form.append(k, v));
form.append("file", theFile);
await fetch(up.url, { method: "POST", body: form });
// 204 = ok · 400 (EntityTooLarge) = the file was over the limit 🛑
```

## Measure a folder: `s3.usage` 📦

Want to know how much storage one folder is using — say everything under `user-a/`?
`s3.usage` adds it all up for you.

```js
function handler(ctx) {
  var u = s3.usage({ prefix: "user-a/" });
  // u = { prefix: "user-a/", bytes: 5242880, objects: 137 }
  return json({ megabytes: u.bytes / 1048576, files: u.objects }, null);
}
```

A few things to know about this one:

- **It's the _only_ button here that actually phones your storage.** Every other `s3`
  call is pure math; `usage` signs a listing request and reads the answer back (it still
  goes through the same safety guard on the `endpoint`). Leave the `prefix` off to total
  the **whole bucket**.
- **There are no real "folders" in S3.** `user-a/` is just the start of a key name, and
  there's no single "how big is this folder" command anywhere — not even in the AWS
  console. The only way to know is to walk every file under the prefix and add up the
  sizes, which is exactly what `usage` does (1000 files at a time, following the trail to
  the next page until it's done).
- **Big folders cost more ops.** Each page of 1000 files counts as one operation against
  your per-run limit (`max_ops`). A folder with millions of files will hit that limit and
  stop with an error rather than run forever. For folders that big, keep your own running
  total in a database (the `db` capability) and use `usage` now and then to re-check it.

## The general form: `s3.presign`

```js
s3.presign({ method: "PUT", key: "uploads/x.txt", expires: 120 });
// method can be "PUT", "GET", "HEAD", or "DELETE" (defaults to "PUT")
```

## When something goes wrong

`s3.presign*` **throws an error** if something is missing (like an empty `key`).
Catch it with `try/catch`:

```js
function handler(ctx) {
  try {
    var put = s3.presignPut({ key: ctx.key });
    return json({ url: put.url }, null);
  } catch (e) {
    return json(null, { message: e.message });
  }
}
```

## It shows up on the receipt 🧾

Each signed link is listed in `meta.s3_requests` (the method and how long it lasts).
